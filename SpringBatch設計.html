1. 本書の目的

本書は、AWS EKS 上で稼働する Spring Batch アプリケーションの設計方針を定めるものである。バッチ開発者およびインフラ構築担当者が共通の設計指針に基づき、ジョブの実行方式・リソース構成・DB 接続数の見積もり等を実装に反映できることを目的とする。

3. システム構成

3.1 構成概要

本システムは、AWS EKS 上で稼働する Spring Boot アプリケーション群と、JP1 によるジョブスケジュール管理を組み合わせた構成である。JP1 はバッチサーバー(EC2)に対してジョブ実行を指示し、バッチサーバーが EKS に対してジョブ投入を行う中継役を担う。

3.2 システム構成図

3.3 構成要素

(1) JP1 (ジョブスケジューラ)

ジョブの実行スケジュール、順序、ジョブネットを管理する。バッチサーバーに対して JP1 エージェント経由でジョブ実行を指示し、シェルスクリプトの戻り値(終了コード)でジョブの成否を判定する。

(2) バッチサーバー (中継役)

配置EKS と同一 VPC 内の EC2
冗長化Active-Standby 構成
役割JP1 と EKS の中継、ジョブ投入ロジックの集約
インストールJP1 エージェント、kubectl、AWS CLI
認証EC2 にアタッチした IAM ロール経由で EKS にアクセス
主要処理Job 名採番、Job マニフェスト生成、kubectl apply、完了待機、ログ取得、クリーンアップ

(3) EKS クラスタ

配置バッチサーバーと同一 VPC
エンドポイントプライベートエンドポイント (VPC 内からのみアクセス可)

(4) Aurora (PostgreSQL)

業務データおよびジョブ要求テーブル、Spring Batch のメタデータテーブルを格納する。EKS 上の各 Pod から JDBC 接続で利用する。

3.6 ネットワーク構成

  • バッチサーバー、EKS クラスタ、Aurora は同一 VPC に配置する。
  • EKS API エンドポイントはプライベートエンドポイントとし、VPC 内からのみアクセス可能とする。
  • セキュリティグループにより、バッチサーバー → EKS API、EKS Pod → Aurora の通信のみ許可する。

4. EKS の基本概念

本章では、Spring Batch のジョブ実行方式を理解するうえで必要となる EKS (Kubernetes) の基本概念を整理する。

4.1 Kubernetes リソース

Kubernetes (EKS) における「リソース」とは、Kubernetes クラスタ上で管理される構成要素の単位を指す。アプリケーションの実行単位 (Pod) や、その起動・停止・スケジュール等を制御する管理オブジェクトを総称してリソースと呼ぶ。

リソースは YAML ファイル (マニフェスト) で「あるべき状態」を宣言的に定義し、Kubernetes がその状態を維持するように動作する。開発者・運用者は手続き的に「起動する／停止する」と命令するのではなく、「この状態であってほしい」と宣言することで Kubernetes に管理を委ねる。

4.2 構成概念 (Cluster / Node / Pod / Container)

EKS の物理層から論理層までの構成要素を整理すると以下の通りとなる。

(1) Cluster (クラスタ)

EKS の最上位の管理単位。複数の Node を束ねて 1 つの実行基盤として扱う。クラスタ単位でネットワーク・認証・名前空間 (Namespace) が管理される。

(2) Node (ノード)

クラスタを構成する物理的なサーバー。EKS においては EC2 インスタンス (または Fargate) が Node の実体となる。Node の上で複数の Pod が動作する。Node の管理 (起動・停止・スケール) は Kubernetes と AWS が自動的に行う。

(3) Pod (ポッド)

コンテナを実行するための最小単位であり、Kubernetes が管理する基本オブジェクト。1 つの Pod には通常 1 つのコンテナを格納する (複数のコンテナを格納することも可能)。Pod は使い捨てを前提とした存在であり、障害発生時は再生成され、IP アドレスや Pod 名は変動する。

(4) Container (コンテナ)

Pod の中で実際に動作するアプリケーション本体。Docker イメージから生成され、Spring Boot アプリケーションはこのコンテナ内で動作する。

(5) 各概念の対応関係

概念役割寿命の傾向本システムでの実体
Cluster実行基盤全体長期EKS クラスタ
Nodeコンテナを動かすサーバー長期 (数ヶ月〜年)EC2 インスタンス
Podコンテナの実行単位短期 (使い捨て)Spring Boot を内包する単位
Containerアプリケーション本体短期 (Pod に従属)Spring Boot アプリ

4.3 リソースの階層構造

Pod はコンテナを実行する最小単位であり、管理リソースは Pod のライフサイクル (起動・終了・再起動・スケジュール) を制御する役割を担う。アプリケーション開発者が直接 Pod を作成することはなく、必ず管理リソース経由で Pod を生成する。

4.4 主要リソース

(1) Deployment

常駐型のアプリケーションを管理するためのリソース。指定した数の Pod を常時稼働状態で維持し、Pod が異常終了した場合は自動的に再起動する。Web アプリケーションおよび常駐型バッチで使用する。主な特徴は、レプリカ数の維持、ローリングアップデートによる無停止デプロイ、Pod 障害時の自動復旧である。

(2) Job

1 回限りのバッチ処理を管理するためのリソース。Pod を起動し、処理が正常終了 (exit code 0) した時点で完了とみなす。失敗時は backoffLimit で指定した回数までリトライを行う。都度起動型バッチで使用する。Deployment と異なり、Pod の正常終了を「障害」ではなく「完了」として扱うため、有限の処理を持つバッチジョブに適している。

(3) CronJob

スケジュールに基づいて Job を生成するリソース。cron 形式のスケジュール定義により、定時起動のバッチを実現する。

補足本システムではジョブスケジュール管理を JP1 で行うため、CronJob は使用しない。

4.5 Pod

Pod はコンテナを動かす最小単位である。Spring Boot のコンテナ (Docker イメージ) は Kubernetes 上で直接実行されるのではなく、Pod という「入れ物」に包まれて動作する。

4.6 replicas (レプリカ)

replicas は「同じ Pod を何個起動しておくか」を指定する数である。複数 Pod を管理するリソースのマニフェスト (YAML) で指定する。

リソースreplicas 指定用途
Deploymentあり常駐サービス (Web アプリ、常駐バッチ)
StatefulSetあり状態を持つ常駐サービス (DB 等)
ReplicaSetありDeployment の内部で使われる低レベルリソース
Jobなし1 回限りのバッチ (代わりに parallelism / completions を使用)
CronJobなし定時バッチ (内部で Job を生成)
Pod (単体)なし単発の Pod (通常は使用しない)

5. Spring Batch 実行方式 (全体方針)

本システムにおける Spring Batch の全体方針を以下に示す。

  • ジョブのスケジュール管理は JP1 で一元管理する。
  • Spring Batch は AWS EKS のコンテナ上で動作する。
  • 実行方式は「都度起動型」と「常駐型」の 2 種類とする。
方式概要
都度起動型 ジョブ投入時に必要に応じてコンテナを起動する。JP1 から決められた時刻・順序でジョブを投入する。1 ジョブ = 1 Kubernetes Job = 1 Podとし、ジョブごとにパラメータ (Job 名、起動引数等) を埋め込んだ Job マニフェストを生成して投入する。
常駐型 コンテナを常時起動しておき、Spring の @Scheduled で動作するジョブがジョブ要求テーブルを参照する。要求があれば Spring Batch ジョブを起動する。

5.1 共通方針

ログ出力コンテナ管理のため標準出力に出力し、Dynatrace で集約する。
DB コネクションAurora の max_connections と Pod 数 × HikariCP プールサイズの掛け算に注意する (詳細は第 8 章)。
メタデータテーブルSpring Batch の BATCH_* テーブルは H2 DB に作成するが、業務では利用しない。ジョブ管理は JP1 と PostgreSQL 上の独自テーブルで行う。
シークレット管理DB パスワード等は AWS Parameter Store で管理する。

6.2 設計ポイント

  • 同時実行されるジョブの最大数は 32 個を前提とする。
  • Job の restartPolicy は Never (または OnFailure) とする。
  • backoffLimit でリトライ回数を制御する。
  • ttlSecondsAfterFinished で完了後の Pod を自動削除する (履歴管理は別途行う前提)。
  • Spring Batch 側は spring.batch.job.enabled=true で起動時にジョブ実行 → 完了で JVM 終了。
  • System.exit(SpringApplication.exit(ctx)) で終了コードを正しく返却する。

7.3 設計ポイント

(1) replicas は基本 1 とする

複数 Pod が同じジョブ要求テーブルを参照すると、同一ジョブの二重起動が発生し得る。冗長化を行う場合は、ShedLock または DB の SELECT ... FOR UPDATE SKIP LOCKED による排他制御を導入する。

(2) ジョブ実行は別スレッドで行う

@Scheduled のスレッドでそのまま Spring Batch を起動すると、長時間ジョブ実行中にポーリングが停止する。TaskExecutor (ThreadPoolTaskExecutor) に処理を委譲するか、JobLauncher を非同期化する。

(3) Graceful Shutdown を必須とする

Pod 再起動時にジョブが中断されないよう、SIGTERM を受信した場合は実行中ジョブの完了を待つ実装とする。terminationGracePeriodSeconds もジョブの最大実行時間に合わせて長めに設定する。

(4) ジョブ要求テーブルの排他制御を実装する

replicas=1 であっても、デプロイ中 (Rolling Update) には新旧 2 Pod が並走する瞬間がある。同一ジョブ要求を 2 回拾わないよう、以下のいずれかの方式で排他制御を行う。

方式概要
対応例 1：
SELECT ... FOR UPDATE SKIP LOCKED		 SELECT 時に DB の行ロックを取得し、他トランザクションがロック中の行は自動でスキップして次の行を読む方式。PostgreSQL 標準機能で実現でき、LIMIT N で複数件をまとめて並行取得しやすい。
対応例 2：
ステータスカラムを
UPDATE 戻り値で排他制御		 UPDATE ... SET status='実行中' WHERE id=? AND status='待機' を発行し、戻り値の更新行数が 1 なら獲得成功、0 なら他 Pod に先取りされたと判定する方式。実装がシンプルで、updated_by 等のカラムで「どの Pod が処理中か」を可視化できるため運用調査がしやすい。

(5) 冗長化

Active-Standby 構成とする (実体としては replicas=1 で運用し、排他制御により重複実行を防止する)。

8.2 プールサイズ計算 (リトルの法則)

Web アプリのプールサイズは、待ち行列理論の「リトルの法則」に基づき算出する。

(1) 基本式

必要プールサイズ = 1秒あたりのリクエスト数 × 1リクエストあたりの平均DB占有時間

(2) Web アプリへの適用

1リクエストあたりのDB占有時間
 = DB処理回数 × 1処理あたりの平均時間
 = 3回 × 0.03秒 (30ms)
 = 0.09秒

1Podあたり同時に必要な接続数
 = 1Podあたりのリクエスト数 × DB占有時間
 = 146 req/秒 × 0.09秒
 = 約 13.1 接続

ピーク時バッファ (×2倍)
 = 13.1 × 2
 = 約 26 接続

安全マージン込みの推奨値
 = 30〜40 接続  → 採用値: 35

(3) プールサイズ設定の注意点

  • プールサイズが大きすぎると DB 側のコンテキストスイッチ増加・メモリ消費増・ロック競合増を招く。
  • HikariCP 公式は「多くのアプリで 10 接続でも十分」としており、過剰設定を避けること。
  • 最終値は本番前の負荷試験で hikaricp_pending_connections (接続待ち数) を監視し、実測ベースで調整する。

8.4 Aurora インスタンスサイズの推奨

Aurora PostgreSQL の max_connections はインスタンスクラスのメモリサイズに連動して自動設定される (パラメータグループで LEAST({DBInstanceClassMemory/9531392},5000) がデフォルト)。

インスタンスクラスメモリmax_connections (目安)約 914 接続に対する評価
db.t4g.medium4 GB約 450不可接続数が不足
db.r6g.large16 GB約 1,800要検証接続数は OK だが性能面で要検証
db.r6g.xlarge32 GB約 3,600推奨本構成の推奨インスタンス
db.r6g.2xlarge64 GB約 7,200余裕大幅余裕

70,000 req/分のピーク負荷かつ複雑な JOIN を伴う処理を捌くため、接続数の観点だけでなく CPU/メモリ性能の観点からも db.r6g.xlarge 以上を推奨する。参照系クエリの負荷分散を行う場合はライター + リーダー構成も検討する。

8.5 設計時の注意点

(1) 接続プールサイズの設定方針

  • HikariCP maximum-pool-size は「同時に必要な最大接続数」をベースに設定する。大きすぎると Aurora 側が枯渇し、小さすぎると業務側で接続待ちが発生する。
  • バッチ処理の並列度 (Spring Batch の taskExecutor のスレッド数) と HikariCP プール数を一致させること。

(2) Spring Batch メタデータ接続の考慮

  • Spring Batch は JobRepository 用に別途接続を消費する。業務用と同じ DataSource を使う場合はプール数を多めに、別 DataSource を切る場合は専用プールを定義する。

(3) 都度起動型 Job の同時実行数制御

重要本システムは都度起動 Job を 最大 32 本同時実行を前提として設計しており、DB 接続数の大部分 (480 接続) を消費する区分となる。JP1 側で同時投入数を 32 以下に制御するか、Kubernetes の ResourceQuota / Job の parallelism で制御する。無制限に Job を起動すると Aurora の接続数が枯渇するリスクがある。

(4) Tomcat スレッド数とプールサイズの関係

  • Tomcat スレッド数 ≧ HikariCP プールサイズ となるよう設計する。
  • スレッド数 200 に対しプール 35 の場合、最大 165 スレッドは DB 接続待ちとなり得るが、これはプール枯渇を防ぐための正常動作である。応答時間への影響は負荷試験で実測すること。

(5) 監視項目

  • CloudWatch メトリクス：DatabaseConnections、MaxDatabaseConnectionsAllowed、CPUUtilization
  • HikariCP メトリクス：hikaricp_connections_active、hikaricp_connections_pending
  • アラート閾値：max_connections の 70% 到達で警告、85% で重大
