WSL からの実行に関する前提事項

本プロジェクトのシェルスクリプトは Windows 上の WSL2（Ubuntu） から実行します。JAR ファイルは Windows 側の Eclipse でビルドし、WSL から Windows のファイルシステムをマウントして実行します。

📁
WSL からの Windows ファイルアクセス

WSL では Windows の C:\ ドライブが /mnt/c/ としてマウントされます。Eclipseのワークスペースは WSL から以下のパスでアクセスできます。

# Windows パス
C:\pleiades\2025-09\workspace_job_sample\BatchSample\

# WSL からのパス
/mnt/c/pleiades/2025-09/workspace_job_sample/BatchSample/
⚠️
WSL から Windows の PostgreSQL へ接続するための設定

WSL（Linux）から Windows 側の PostgreSQL に接続するには、以下の設定が必要でした。デフォルトでは WSL からの接続は拒否されます。

設定ファイル / 手順変更内容理由
postgresql.conf listen_addresses = '*' に変更 デフォルトは localhost のみ待受。全IPからの接続を許可するため。
pg_hba.conf host all all 0.0.0.0/0 scram-sha-256 を追加 WSL の IP からの接続を認証付きで許可するため。
接続先 IP の取得 WSL 内で ip route | grep default を実行 WSL から Windows ホストへの接続には nameserver ではなく default gateway の IP（172.xx.x.x 系）を使用する必要があるため。

シェルスクリプト内で以下のコマンドにより実行時に自動取得しています。

WINDOWS_HOST=$(ip route | grep default | awk '{print $3}')

プロジェクト概要

本プロジェクト BatchSample は、Spring Boot 3.5.14 と Spring Batch 5 を用いて構築したバッチ処理基盤のサンプル実装です。1つのJARファイルに以下の3つの役割を統合し、起動引数によって動作モードを切り替えます。

# 役割 ジョブ名 起動方法
① 常駐スケジューラー（ジョブ要求監視） — 引数なしで起動
② JP1から直接起動されるバッチジョブ helloJP1Job ジョブ名を引数で指定
③ スケジューラーから起動されるバッチジョブ helloScheduledJob ①のスケジューラーが自動起動

Spring Batch メタテーブルについて

Spring Batch は実行履歴を管理するためのメタテーブル（BATCH_JOB_INSTANCE、BATCH_JOB_EXECUTION、BATCH_STEP_EXECUTION 等）を使用します。本プロジェクト（開発環境）では spring.batch.jdbc.initialize-schema=always を設定し、PostgreSQL に自動生成させています。ジョブのリラン制御は RunIdIncrementer により対応しています。

🚨
【本番環境への注意事項】initialize-schema は必ず never に変更すること

spring.batch.jdbc.initialize-schema=always のまま本番環境で運用すると、起動のたびにメタテーブルの再作成が試みられ、既存の実行履歴データが失われるリスクがあります。

本番環境では以下の手順に従うこと：

  1. 本番 PostgreSQL に対して Spring Batch の DDL スクリプト（schema-postgresql.sql）を使用し、メタテーブルを手動で作成する。
  2. application-prod.yml に spring.batch.jdbc.initialize-schema: never を設定する。
  3. テーブルが存在しない状態で never を指定すると起動時エラーになるため、必ずテーブル作成後に設定すること。
# application-prod.yml
spring:
  batch:
    jdbc:
      initialize-schema: never  # 本番では必ず never！

主要アノテーション解説

本プロジェクトの要件に直接関係するアノテーションを解説します。

アノテーション 付与クラス 役割・説明
@EnableScheduling BatchSampleApplication Spring のスケジューリング機能をアプリケーション全体で有効化する。このアノテーションがないと @Scheduled は一切動作しない。アプリ内で1箇所だけ付与すれば十分。
@Scheduled JobScheduler メソッドを定期実行するアノテーション。fixedDelay = 5000 は前回実行の完了から5秒後に次の実行を開始する。fixedRate と異なり処理の重複が発生しないため、バッチ処理に適している。
@Qualifier JobScheduler 同じ型（Job）の Bean が helloJP1Job と helloScheduledJob の2つ存在するため、@Autowired 時にどちらを注入するかを名前で明示的に指定する。指定しないと Spring が注入先を判断できずエラーになる。
@ConditionalOnProperty JobScheduler 指定したプロパティ値に応じて Bean の生成自体を制御する。scheduler.enabled=false を渡すと JobScheduler Bean が生成されず、@Scheduled が動かなくなる。JP1バッチモードでスケジューラーを止めるために使用。
@Configuration BatchConfig Spring の設定クラスであることを示す。@Bean メソッドでジョブ・ステップを定義する。
@Component HelloTasklet 他 Spring の DI コンテナにコンポーネントとして登録する。@Autowired による自動注入が可能になる。
@Slf4j JobScheduler 他 Lombok が提供するアノテーション。クラスに付与するだけで log.info() / log.error() 等が使えるようになる。

ジョブ定義

helloJP1Job（JP1起動ジョブ）

項目値
ジョブ名helloJP1Job
ステップ名helloJP1Step
TaskletHelloTasklet
出力メッセージ===== Hello from JP1 Job! =====
起動方法シェル引数でジョブ名指定
インクリメンターRunIdIncrementer

helloScheduledJob（スケジューラー起動ジョブ）

項目値
ジョブ名helloScheduledJob
ステップ名helloScheduledStep
TaskletHelloScheduledTasklet
出力メッセージ===== Hello from Scheduled Job! =====
起動方法JobScheduler が5秒ごとに自動起動
インクリメンターRunIdIncrementer

BatchConfig.java — ジョブ・ステップ定義コード

BatchConfig.java
@Configuration
public class BatchConfig {

    // ② JP1から起動されるジョブ
    @Bean
    public Step helloJP1Step(JobRepository jobRepository,
                              PlatformTransactionManager transactionManager,
                              HelloTasklet helloTasklet) {
        return new StepBuilder("helloJP1Step", jobRepository)
                .tasklet(helloTasklet, transactionManager)
                .build();
    }

    @Bean
    public Job helloJP1Job(JobRepository jobRepository, Step helloJP1Step) {
        return new JobBuilder("helloJP1Job", jobRepository)
                .incrementer(new RunIdIncrementer())
                .start(helloJP1Step)
                .build();
    }

    // ③ @Scheduled から起動されるジョブ
    @Bean
    public Step helloScheduledStep(JobRepository jobRepository,
                                    PlatformTransactionManager transactionManager,
                                    HelloScheduledTasklet helloScheduledTasklet) {
        return new StepBuilder("helloScheduledStep", jobRepository)
                .tasklet(helloScheduledTasklet, transactionManager)
                .build();
    }

    @Bean
    public Job helloScheduledJob(JobRepository jobRepository, Step helloScheduledStep) {
        return new JobBuilder("helloScheduledJob", jobRepository)
                .incrementer(new RunIdIncrementer())
                .start(helloScheduledStep)
                .build();
    }
}

起動方法（コマンドレベル）

常駐スケジューラーモード（引数なし）

ジョブ要求テーブルを監視し続ける常駐プロセスとして起動します。@Scheduled が有効になり、5秒ごとに helloScheduledJob が実行されます。

コマンド
./run_batch.sh
内部で実行される java コマンド
java -jar BatchSample.jar \
  --spring.profiles.active=dev \
  --spring.datasource.url=jdbc:postgresql://172.24.0.1:5432/batch_db \
  --spring.batch.job.enabled=false \
  --scheduler.enabled=true

各オプションの背景・理由

--spring.batch.job.enabled=false
Spring Boot はデフォルトで起動時にコンテキスト内の全 Job Bean を自動実行しようとします。常駐スケジューラーモードでは @Scheduled によって Job を制御したいため、起動時の自動実行を明示的に無効化しています。これがないと起動直後に全 Job が実行されてしまいます。
--scheduler.enabled=true
JobScheduler クラスに @ConditionalOnProperty(name = "scheduler.enabled", havingValue = "true", matchIfMissing = true) を付与しているため、このプロパティが true のときだけ JobScheduler Bean が生成されます。省略時のデフォルトも true（matchIfMissing = true）のため、常駐モードでは明示的に指定しています。
--spring.profiles.active=dev
application-dev.yml の DB 接続設定を読み込むためのプロファイル指定です。本番では JP1 の環境変数 SPRING_PROFILES_ACTIVE=prod で切り替えます。
--spring.datasource.url=jdbc:postgresql://172.24.0.1:5432/batch_db
WSL から Windows 側の PostgreSQL に接続するために、Windows ホストの IP アドレスを指定しています。localhost では WSL 自身を指してしまい接続できません。IP はシェルスクリプト内で ip route | grep default コマンドにより自動取得しています。

JP1バッチモード（ジョブ名指定）

指定したジョブを1回だけ実行して終了します。スケジューラーは起動しません。

コマンド
./run_batch.sh helloJP1Job
内部で実行される java コマンド
java -jar BatchSample.jar \
  --spring.profiles.active=dev \
  --spring.datasource.url=jdbc:postgresql://172.24.0.1:5432/batch_db \
  --spring.batch.job.enabled=true \
  --spring.batch.job.name=helloJP1Job \
  --scheduler.enabled=false

各オプションの背景・理由

--spring.batch.job.enabled=true
起動時に指定したジョブを自動実行するために true を指定します。false のままだと JAR は起動するだけでジョブが実行されず、そのまま終了してしまいます。
--spring.batch.job.name=helloJP1Job
JAR の中には複数の Job Bean（helloJP1Job、helloScheduledJob）が含まれています。このオプションで実行対象のジョブ名を明示的に指定しないと、全ての Job が実行されてしまいます。JP1 からはジョブ名を引数としてシェルに渡し、シェルがこのオプションに設定します。
--scheduler.enabled=false
JAR には @EnableScheduling が付いているため、JP1バッチモードで起動しても @Scheduled のスレッドが同時に起動してしまいます。scheduler.enabled=false を渡すことで @ConditionalOnProperty の条件が外れ、JobScheduler Bean 自体が生成されなくなります。これにより helloScheduledJob が意図せず実行されることを防ぎます。

本番環境（JP1）での起動例

JP1 環境変数設定 → シェル呼び出し
# JP1 側で環境変数を設定
SPRING_PROFILES_ACTIVE=prod
SPRING_DATASOURCE_URL=jdbc:postgresql://本番DBホスト:5432/batch_db

# JP1 からシェルを呼び出す
/app/shell/run_batch.sh helloJP1Job

設計ポイント・注意事項

⚠️
--spring.batch.job.enabled=true を渡すと @Scheduled も同時に動く

Spring Boot は @EnableScheduling が付いている場合、JARの起動と同時に @Scheduled のスレッドも開始します。JP1バッチモードで --spring.batch.job.enabled=true を渡しても JobScheduler の @Scheduled は別スレッドで動いてしまいます。必ず --scheduler.enabled=false とセットで指定してください。

⚠️
同一ジョブ名＋同一パラメータでのリラン問題

Spring Batch は同じジョブ名＋同じパラメータの組み合わせで COMPLETED になると再実行を拒否します（エラー：Job instance already exists and is complete）。全ジョブに RunIdIncrementer を設定し、実行ごとに自動的にユニークな run.id パラメータを付与することで対応しています。

💡
将来拡張：ジョブ要求テーブルによる動的ジョブ起動

現在の JobScheduler は helloScheduledJob を直接 @Qualifier で指定していますが、将来的にはジョブ要求テーブルからジョブ名を動的に取得し、ApplicationContext.getBean(jobName, Job.class) で Job を動的に解決する設計に拡張します。この場合 @Qualifier は不要になります。

✅
Spring Batch 5 での変更点（JobBuilderFactory 廃止）

Spring Batch 5 では JobBuilderFactory および StepBuilderFactory が廃止されました。代わりに JobBuilder と StepBuilder を直接使用し、JobRepository を第2引数として渡します。また Spring Boot 3.x では @EnableBatchProcessing は不要です。

run_batch.sh（完全版）

shell/run_batch.sh
#!/bin/bash

# WindowsホストのIPをip routeから自動取得（WSL環境用）
# 本番（Linux）ではこの行は不要
WINDOWS_HOST=$(ip route | grep default | awk '{print $3}')

JAR_PATH="/mnt/c/pleiades/2025-09/workspace_job_sample/BatchSample/target/BatchSample.jar"

# プロファイル設定
# ローカル：以下の変数で切り替える
# 本番：JP1の環境変数 SPRING_PROFILES_ACTIVE で定義すること
SPRING_PROFILES_ACTIVE=${SPRING_PROFILES_ACTIVE:-dev}

# DB接続URL
# ローカル：WSLからWindowsホストのIPを自動取得して設定
# 本番：JP1の環境変数 SPRING_DATASOURCE_URL で定義すること
SPRING_DATASOURCE_URL=${SPRING_DATASOURCE_URL:-jdbc:postgresql://${WINDOWS_HOST}:5432/batch_db}

# 起動モードの判定
# 引数なし  → 常駐スケジューラーモード（@Scheduled が動き続ける）
# 引数あり  → JP1バッチモード（指定ジョブを1回実行して終了）
#             注意：scheduler.enabled=false を必ず指定すること！
#                   指定しないと @Scheduled も同時に動いてしまう！
if [ -z "$1" ]; then
  echo "===== スケジューラー起動 profile=[${SPRING_PROFILES_ACTIVE}] ====="
  java -jar ${JAR_PATH} \
    --spring.profiles.active=${SPRING_PROFILES_ACTIVE} \
    --spring.datasource.url=${SPRING_DATASOURCE_URL} \
    --spring.batch.job.enabled=false \
    --scheduler.enabled=true
else
  JOB_NAME=$1
  echo "===== バッチ開始 [${JOB_NAME}] profile=[${SPRING_PROFILES_ACTIVE}] ====="
  java -jar ${JAR_PATH} \
    --spring.profiles.active=${SPRING_PROFILES_ACTIVE} \
    --spring.datasource.url=${SPRING_DATASOURCE_URL} \
    --spring.batch.job.enabled=true \
    --spring.batch.job.name=${JOB_NAME} \
    --scheduler.enabled=false
  echo "===== バッチ終了 [${JOB_NAME}] ====="
fi
