1. 事前準備

1.1 日本語化

1
VSCodeを起動し、拡張機能パネルを開く（Ctrl+Shift+X）。
2
検索ボックスに Japanese Language Pack と入力し、インストールをクリック。
3
インストール完了後、ダイアログの「Restart」をクリックしてVSCodeを再起動。
4
UIが日本語表示に切り替わっていれば完了。

1.2 拡張機能のインストール

以下の拡張機能を拡張機能パネル（Ctrl+Shift+X）からインストールしてください。

拡張機能名用途区分
Japanese Language Pack for Visual Studio CodeVSCode UIの日本語化必須
GitHub Copilot ChatAIチャット・コード補完・Agentモード（全機能統合）必須
Extension Pack for JavaJava構文認識・補完・デバッグ推奨
Spring Boot Extension PackSpring Boot解析・設定支援推奨
⚠ 注意
旧来の拡張機能「GitHub Copilot」は非推奨です。GitHub Copilot Chat がコード補完・チャット・Agentのすべての機能を統合しており、こちらのみのインストールで問題ありません。

2. GitHub Copilotの基本設定

2.1 GitHub Copilotへのサインイン

1
VSCode左下の人型アイコンをクリック → 「GitHub でサインイン」を選択。
2
ブラウザが起動するのでGitHubにログインし、「Authorize Visual Studio Code」をクリック。
3
「VSCodeを開く」の確認ダイアログで許可。
4
VSCodeのステータスバーにCopilotアイコンが表示されれば完了。

有効化の確認URL：https://github.com/settings/copilot

2.2 gitのローカルリポジトリをワークスペースとして開く

1
メニュー「ファイル(F)」→「フォルダーを開く」をクリック。
2
リポジトリのルートフォルダを指定する（例：C:\Users\yourname\git\springboot-demo）。
3
VSCodeの左下にブランチ名（例：main）が表示されれば、Gitリポジトリとして認識された状態。
ⓘ ポイント
ワークスペースとは、VSCodeで開いているフォルダそのものです。フォルダ内の全ファイル・サブフォルダが作業対象になり、Copilotの @workspace コマンドでフォルダ全体を解析できます。Eclipseのように専用の設定フォルダが別の場所にあるわけではなく、開いたフォルダ＝ワークスペースです。

3. GitHub Copilotの機能概要

3.1 チャットモード

💬 Ask モード

1問1答形式。コードの説明・質問・ドキュメント生成に最適。消費コントロールがしやすい。

✍ Edit モード

指定したファイルを直接編集する。変更点をDiff形式で確認・適用できる。

🤖 Agent モード

計画立案・ファイル読込・コマンド実行・コード修正を自律的に繰り返す。複雑タスク向け。

📋 Plan モード

実装計画をまず立案し、内容確認後に実行する。大きな変更を安全に行いたい場合に有効。

3.2 プレミアムリクエストの仕組み

Copilotの利用はリクエスト数でカウントされ、利用するモデルや機能によって「プレミアムリクエスト」の消費量が変わります。

モード消費の特徴注意点
Ask モード
（標準チャット）		 1ユーザーメッセージ ＝ 1プレミアムリクエスト
（選択モデルの倍率を乗算）		 消費量をコントロールしやすく、予期せぬ大量消費が起きにくい。
Agent モード
（自律実行）		 ユーザー入力1回でも、内部の「プランニング」「ツール実行」「ステップ推論」が個別リクエストとしてカウントされる。 複雑なタスクや @workspace 等の広範なコンテキスト読み込み・リトライが発生すると消費が激増しやすい。意識的な利用を推奨。

3.3 利用可能なAIモデルとプレミアムリクエスト

使用するモデルによってプレミアムリクエストの消費有無や倍率が異なります。目的に応じてモデルを選択してください。

モデル特徴おすすめ用途プレミアム消費
GPT-4o バランス型・高速 一般的なチャット・コード補完 なし（included）
GPT-4.1 高精度コード生成 コードレビュー・リファクタリング ⭐ なし（included）
GPT-5 mini 軽量・高速 簡単な質問・補完 なし（included）
Claude Sonnet 4.6 文章要約・説明が得意 仕様書作成・要約・コードレビュー ⭐ あり（倍率あり）
Claude Opus 4.7 Anthropic最上位・長期推論・複雑なエージェント実行に強い 複雑なアーキテクチャ設計・難解バグ解析・長時間Agentタスク ⭐ あり（高倍率：15×）
Gemini 2.5 Pro 長いコンテキスト処理 大規模ファイル解析・@workspace活用 あり（倍率あり）
GPT-5 / GPT-5.5 最高性能・複雑な推論 複雑なアーキテクチャ設計・難解バグ解析 あり（高倍率）
ⓘ コードレビュー・リファクタリングにおすすめのモデル
GPT-4.1（プレミアム消費なし・高精度なコード生成）と Claude Sonnet 4.6（説明文の質が高く、レビューコメントが分かりやすい）が特におすすめです。高度な推論が必要な場合は Claude Opus 4.7 も選択肢になりますが、消費倍率（15×）が高い点に注意してください。
⚠ 重要：2026年6月1日より課金方式が変更されます
GitHubは2026年6月1日より、Copilotの課金方式をリクエストベースから使用量ベース（Usage-based billing）に変更します。個人・組織・エンタープライズのすべてのアカウントが対象です。

・組織・エンタープライズ向け詳細：usage-based-billing-for-organizations-and-enterprises
・個人向け詳細：usage-based-billing-for-individuals

変更前に公式ドキュメントを確認し、利用状況を把握しておくことを推奨します。

プレミアムリクエストの詳細説明：https://docs.github.com/en/copilot/concepts/billing/copilot-requests

5. GitHub Copilot Chat の指示を共通化する方法

毎回のチャットで同じ前提条件や規約を伝えるのは手間がかかります。Copilotでは設定ファイルを使って指示を自動化・共通化する仕組みが用意されており、一度設定すれば指示が自動的にCopilotへ渡されます。利用シーンに応じて設定ファイルを使い分けることができます。

⚠ 重要な注意点：コンテキストの消費について
共通化した指示ファイル（copilot-instructions.md・*.instructions.md・AGENTS.md 等）の内容は、毎回のチャットリクエストのたびに自動的にAIへ送信されます。これはAIが使用できる「コンテキストウィンドウ（一度に処理できる情報量）」を消費することを意味します。

指示が長くなるほど、実際のコードや会話のやりとりに使えるコンテキストが圧迫されます。また、プレミアムリクエストの消費量にも影響する場合があります。指示は簡潔・要点のみに絞り、不要な情報は書かないことを推奨します。
ⓘ 全体の優先順位
複数の設定ファイルが同時に存在する場合、すべての指示がCopilotに渡されます。競合がある場合は「個人設定（ユーザーレベル）」が最優先、次に「リポジトリ設定」、最後に「組織設定」の順で適用されます。

5.1 各設定ファイルの概要と使い分け

① .github/copilot-instructions.md — リポジトリ全体への常時適用指示
保存場所：<リポジトリルート>/.github/copilot-instructions.md　｜　適用：自動（常時）　｜　スコープ：リポジトリ全体

概要：リポジトリ内のすべてのチャットリクエストに自動的に付加される、最も基本的な共通指示ファイルです。このファイルの内容は毎回のチャット送信時にコンテキストとして消費されるため、内容は簡潔に保つことが重要です。

適している内容：

  • コーディングスタイルや命名規則（例：メソッド名はキャメルケース）
  • 使用技術スタックの宣言（例：Java 17 / Spring Boot 3.x / Maven）
  • アーキテクチャ上の方針（例：サービス層にビジネスロジックを集約する）
  • セキュリティ要件や例外処理の方針
  • ドキュメントの記述基準

作成方法：リポジトリルートの .github/ フォルダに copilot-instructions.md ファイルを作成し、自然言語（日本語可）で指示を記述するだけです。

# SpringBootプロジェクト共通指示

## コーディング規約
- Java 17の機能（Record, Sealed Class等）を積極的に活用する
- メソッド名はキャメルケース、定数はALL_CAPSで命名する
- すべての公開メソッドにJavadocコメントを付ける

## アーキテクチャ
- ビジネスロジックはServiceクラスに集約する
- ControllerはリクエストのバリデーションとServiceの呼び出しのみ行う

## セキュリティ
- SQLはJPA/Hibernateのクエリメソッドを使用し、生SQLは避ける
② .github/instructions/*.instructions.md — 言語・ファイル別の自動適用指示（ワークスペース）
保存場所：.github/instructions/ フォルダ内（ファイル名末尾は必ず .instructions.md）　｜　適用：自動（ファイル一致時）　｜　スコープ：対象ファイル・フォルダ

概要：ファイルの種類や保存場所に応じて条件付きで自動適用される指示ファイルです。フロントマターの applyTo にglobパターンを記述することで、対象ファイルを操作しているときだけ指示が有効になります。copilot-instructions.md と両方存在する場合は両方の指示が合わせて使われます。

適している内容：

  • フロントエンドとバックエンドで異なる規約がある場合
  • 言語ごとの規約（Javaファイルにはこのルール、HTMLファイルにはこのルール等）
  • テストコード専用の規約（モックの使い方・アサーションの書き方など）

フロントマターの設定例：

---
name: 'Java Standards'
description: 'Javaファイルへのコーディング規約'
applyTo: '**/*.java'
---
# Java ファイル向け規約
- 例外は RuntimeException を継承したカスタム例外クラスで統一する
- Lombokの @Slf4j アノテーションでロガーを生成する
- nullチェックは Optional を使って表現する

globパターン例：**/*.java（全Javaファイル）、src/test/**/*.java（テストのみ）、**/*.html,**/*.css（複数種類の指定）

サブフォルダでの整理例：

.github/instructions/
  backend/
    java.instructions.md       # applyTo: **/*.java
    sql.instructions.md        # applyTo: **/*.sql
  testing/
    unit-test.instructions.md  # applyTo: src/test/**/*.java
  docs/
    markdown.instructions.md   # applyTo: docs/**/*.md
③ ~/.copilot/instructions/*.instructions.md — 個人共通の自動適用指示（全ワークスペース横断）
保存場所：~/.copilot/instructions/（Windows：C:\Users\yourname\.copilot\instructions\）　｜　適用：自動（全WS・ファイル一致時）　｜　スコープ：個人・全プロジェクト共通

概要：ユーザーのホームディレクトリ配下に置く *.instructions.md ファイルです。どのワークスペース（リポジトリ）を開いていても常に読み込まれるため、個人として複数のプロジェクトをまたいで統一したいルールを一元管理できます。ファイル形式・フロントマターの書き方は ② と同じです。

ワークスペースの ② との違い：

  • ② はリポジトリに保存するため、チームで共有・バージョン管理される。
  • ③ は自分のマシン上のみに保存され、個人の作業スタイルに関するルールを管理する。
  • 両方存在する場合はどちらも適用され、③（ユーザーレベル）が優先される。

適している内容：

  • 「コメントは必ず日本語で書く」など個人として統一したい習慣
  • 会社・チーム内の全プロジェクト共通のコーディングポリシー
  • 出力フォーマットの個人的好み（例：箇条書きより散文で説明してほしい）

作成方法：

1
コマンドパレット（Ctrl+Shift+P）→ Chat: New Instructions File を実行。
2
保存先として「User（ユーザープロファイル）」を選択する。
3
ファイル名を入力（例：personal-style）すると ~/.copilot/instructions/personal-style.instructions.md が自動生成される。
4
フロントマターと指示内容を記述して保存すれば即時有効。
---
name: 'Personal Coding Style'
description: '全プロジェクト共通の個人ルール'
applyTo: '**'
---
# 個人共通ルール
- コードコメントは必ず日本語で記述する
- 説明は箇条書きではなく文章形式で出力する
- 提案は必ず理由とセットで示す
④ .github/prompts/*.prompt.md — 手動で呼び出すプロンプトテンプレート
保存場所：.github/prompts/ フォルダ内（ファイル名末尾は必ず .prompt.md）　｜　適用：手動呼び出し　｜　スコープ：任意

概要：自動適用ではなく、必要なときに手動でチャットに呼び出すプロンプトテンプレートです。常時送信されないため、コンテキストを節約しながら定型プロンプトを再利用できます。チャット入力欄で / を入力するとファイル名がサジェストされます。

適している内容：

  • コードレビューの定型チェックリスト
  • 特定の設計パターン（RepositoryパターンやFacadeパターン等）のコード生成テンプレート
  • リリースノートや仕様書の生成テンプレート
  • デバッグ時の調査手順テンプレート

サンプル（.github/prompts/code-review.prompt.md）：

---
name: 'SpringBoot コードレビュー'
description: 'SpringBootのコードを標準観点でレビューする'
---
以下の観点でコードをレビューしてください：

1. **設計**: SOLIDの原則に違反していないか
2. **例外処理**: 適切にtry-catchまたは@ExceptionHandlerで処理されているか
3. **セキュリティ**: 入力値のバリデーションが行われているか
4. **パフォーマンス**: N+1問題やメモリリークのリスクがないか
5. **テスト**: ユニットテストで検証すべき境界値が網羅されているか

問題点は「問題」「理由」「改善案」の形式で出力してください。

呼び出し方：Copilot Chatの入力欄で / を入力 → ファイル名をサジェストから選択 → Enter。または #file:.github/prompts/code-review.prompt.md で参照することも可能です。

⑤ .github/agents/*.agent.md — フェーズ別カスタムエージェント
保存場所：.github/agents/ フォルダ内（ファイル名末尾は必ず .agent.md）　｜　適用：手動選択（ドロップダウン）　｜　スコープ：任意

概要：特定の役割・目的に特化したAIエージェントを定義するファイルです。指示内容だけでなく、使用するモデルや利用可能なツール（ファイル検索、Web取得など）も設定できます。Copilot Chatのモード選択ドロップダウンに表示され、選択するとそのエージェントの設定でチャットが開始されます。詳細は 6. カスタムエージェント を参照してください。

⑥ AGENTS.md — 複数AIツール間での共有指示（オープンスタンダード）
保存場所：リポジトリルート直下 AGENTS.md、またはサブフォルダ内にネスト配置　｜　適用：自動（常時・ツール横断）　｜　スコープ：リポジトリ全体／サブフォルダ

概要：AGENTS.md はOpenAI Codex、Cursor、GitHub Copilot、Claude Code、Google Julesなど複数のAIコーディングツールが共通して読み込めるオープンスタンダードのファイル形式です。チームが複数のAIツールを使い分けている場合に、ルールを1ファイルで一元管理できます。このファイルも毎回のチャット送信時にコンテキストとして消費されます。

copilot-instructions.md との違い：

  • copilot-instructions.md はGitHub Copilot専用。他のAIツールは読み込まない。
  • AGENTS.md はツール横断で有効。チームで複数のAIツールを使う場合に特に有効。
  • 両方存在する場合、GitHub CopilotはどちらもAIへの指示として使用する。

記述内容の例：

# AGENTS.md

## プロジェクト概要
SpringBootを使ったREST APIサーバー。Java 17 / Maven / MySQL 8.0。

## セットアップ
- 依存インストール: `mvn install`
- アプリ起動: `mvn spring-boot:run`
- テスト実行: `mvn test`

## コーディング規約
- コメントは日本語で記述する
- 1クラス1責務の原則を守る
- テストコードはJUnit 5 + Mockitoを使用する

## PRガイドライン
- コミットメッセージは「種別: 内容」形式（例: feat: ログイン機能を追加）
- マージ前に必ず `mvn test` が通ること

モノレポでの使い方：サブフォルダ（例：backend/AGENTS.md、frontend/AGENTS.md）にそれぞれ別の AGENTS.md を置くと、AIエージェントは編集中のファイルに最も近い AGENTS.md を優先して参照します。

公式サイト：https://agents.md

5.2 設定ファイルの使い分けまとめ

ファイル適用タイミングコンテキスト消費主な用途
.github/copilot-instructions.md 自動（常時） 毎回消費 コーディング規約・技術スタックの宣言
.github/instructions/*.instructions.md 自動（ファイル一致時） 一致時に消費 言語別・フォルダ別の細かい規約（チーム共有）
~/.copilot/instructions/*.instructions.md 自動（全WS・ファイル一致時） 一致時に消費 個人共通ルール（プロジェクト横断・自分のみ）
.github/prompts/*.prompt.md 手動呼び出し 呼出時のみ 定型レビュー・生成テンプレート

AGENTS.md 自動（常時・ツール横断） 毎回消費 複数AIツール間の共通指示

6. カスタムエージェント

6.1 カスタムエージェントとは

カスタムエージェントとは、Copilot Chatの動作を独自にカスタマイズした「専用AIアシスタント」です。特定の役割（例：セキュリティレビュー担当、設計書作成担当）に特化した指示・ツール・モデルの組み合わせを定義でき、Copilot Chatのモード選択ドロップダウンから呼び出すだけで、その役割に最適化された状態でチャットが開始されます。

通常のチャットとの違いは、「何をするAIか」を事前に固定できる点です。毎回同じ指示を書く手間がなく、チームで同一のエージェントを共有することで、一貫した品質のアウトプットを得られます。

6.2 カスタムエージェントでできること（具体例）

以下はSpringBootプロジェクト開発で実際に活用できるカスタムエージェントの例です。

コードレビュー
🔍 SpringBoot レビュアー

SOLIDの原則・例外処理・セキュリティ・N+1問題など、一定の観点でコードを自動レビューする。レビューのたびに観点を書く手間がなくなる。

モデル: GPT-4.1
ツール: search/codebase
呼び出し例: 「UserController.java をレビューして」
仕様書・ドキュメント生成
📄 API仕様書ライター

コントローラークラスを読み込み、エンドポイント・リクエスト/レスポンス・エラーコードを一定フォーマットのMarkdown仕様書として出力する。

モデル: Claude Sonnet 4.6
ツール: search/codebase
呼び出し例:「OrderController の仕様書を作成して」
テスト生成
✅ テストコードジェネレーター

サービスクラスの実装を読み取り、JUnit 5 + Mockitoを使ったユニットテストを正常系・異常系・境界値を網羅した形で自動生成する。

モデル: GPT-4.1
ツール: search/codebase
呼び出し例:「UserService のテストコードを生成して」
セキュリティ診断
🔒 セキュリティアナライザー

プロジェクト全体からSQLインジェクション・XSS・認証バイパス・機密情報のハードコードなどのリスクを探し出し、危険度と修正案をセットで報告する。

モデル: Claude Opus 4.7
ツール: search/codebase, web/fetch
呼び出し例: 「セキュリティリスクを全体スキャンして」
設計支援
🛠 アーキテクチャ設計アシスタント

新機能の要件を受け取り、クラス設計・レイヤー分割・テーブル設計・依存関係を提案する。大きな実装を始める前の設計レビューに活用できる。

モデル: Claude Opus 4.7
ツール: search/codebase
呼び出し例: 「注文管理機能の設計案を出して」
リファクタリング
✂ リファクタリングアドバイザー

既存コードの重複・複雑度・命名・責務の混在などを検出し、リファクタリングの優先順位と具体的な改善コードを提示する。

モデル: GPT-4.1
ツール: search/codebase
呼び出し例: 「OrderService をリファクタリングして」
障害調査・デバッグ
📊 デバッグ調査エージェント

スタックトレースやエラーログを貼り付けると、原因箇所の特定・関連コードの参照・修正方法の提案までを一連の流れで実施する。

モデル: GPT-4o
ツール: search/codebase
呼び出し例: 「このスタックトレースの原因を調査して」
DB・SQL支援
📈 SQLレビュアー

リポジトリ内のJPQLやネイティブSQLを読み取り、N+1問題・インデックス欠落・非効率なJOINなどパフォーマンス上のリスクをチェックして改善案を提示する。

モデル: GPT-4.1
ツール: search/codebase
呼び出し例: 「クエリのパフォーマンス問題を診断して」

6.3 カスタムエージェントの構成方法

カスタムエージェントは .agent.md という拡張子のMarkdownファイルで定義します。ファイル冒頭にYAML形式のフロントマターを書き、その後に指示内容（システムプロンプト）を記述します。

■ 保存場所

スコープ保存先パス説明
ワークスペース
（プロジェクト固有）		 .github/agents/
例：C:\Users\yourname\git\springboot-demo\.github\agents\		 このリポジトリを開いたときだけ有効。チームで共有可能（gitでコミットして共有）。フォルダ名 .github/agents はGitHubが定めた固定の規約名であり、リポジトリ構成とは無関係に必ずこの名前にする必要がある。
ユーザープロファイル
（全ワークスペース共通）		 %USERPROFILE%\.copilot\agents\
例：C:\Users\yourname\.copilot\agents\
（Linux/macOS：~/.copilot/agents/）		 どのプロジェクトを開いても常に使えるエージェント。個人設定として管理する。

■ エージェント定義ファイルのサンプル（.github/agents/spring-reviewer.agent.md）

---
name: SpringBootReviewer
description: SpringBootプロジェクトのコードレビューを行う専門エージェント
model: gpt-4.1
tools:
  - search/codebase
  - web/fetch
---

あなたはSpringBootの専門家です。
以下の観点でコードレビューを行ってください。

- Springのベストプラクティスに沿っているか
- 例外処理が適切か
- SQLインジェクションやXSSなどのセキュリティリスクがないか
- 不要な依存関係や冗長なコードがないか

レビュー結果は問題点・理由・改善案の形式で日本語で出力してください。

■ 作成手順

1
コマンドパレット（Ctrl+Shift+P）を開き、「Chat: New Custom Agent」と入力して実行。
2
「Workspace」（プロジェクト固有）または「User」（全体共通）のどちらに保存するか選択。
3
エージェント定義ファイル（.agent.md）が自動生成されるので、指示内容・モデル・ツールを編集して保存。
4
Copilot Chatパネルの入力欄左のドロップダウンに作成したエージェント名が表示されれば完了。

公式ドキュメント：https://code.visualstudio.com/docs/copilot/customization/custom-agents

7.2 Markdownファイルのプレビュー機能

VSCodeにはMarkdownをリアルタイムでプレビューする機能が標準搭載されています。拡張機能なしで利用できます。

操作方法
プレビューを横に開くCtrl + K → V（Kを押したまま離してからV）
プレビューのみ表示Ctrl + Shift + V
右クリックメニューからエディタ上で右クリック → 「プレビューを横に表示」
ⓘ おすすめの使い方
Ctrl+K → V でエディタとプレビューを左右に並べると、Markdownを編集しながらリアルタイムで見た目を確認できます。HTMLファイルはVSCodeの標準機能ではプレビューできないため、Live Server拡張機能（インストール後、右下「Go Live」ボタン）を使うとChromeで自動更新プレビューができます。
